# 更新日志 — 面板信息呈现优化（数据运营视角重构）

---

## 问题诊断

升级前面板存在三个问题：

1. **纯罗列**：所有合并要点平铺直叙，看不出哪些是真正重要的信号
2. **无综合判断**：看完一堆标题，仍需要自己在脑子里归纳"这个板块到底在说什么"
3. **列表过长**：单个分类条目一多，页面被拉得很长，体验像刷一个无止尽的列表

---

## 本次改动（按数据运营常见做法重构）

### 1. 新增「今日要点」摘要层 — 解决"看不出在说什么"

每个分类标题下方新增一个高亮摘要框，用 3-5 条要点直接告诉你
**这个板块的用户/媒体/官方在讨论什么**，而不是让你自己从十几条标题里提炼。

两种生成方式：

- **AI 摘要**（配置 `ANTHROPIC_API_KEY` 后启用）：调用 Claude 真正理解
  本类别下所有合并要点的语义，输出归纳性的中文要点，例如
  "多名UK用户反映NordVPN伦敦节点近期频繁断线"，而不是简单复述标题
- **规则摘要**（默认，无需任何配置）：按重要度提取 Top 3 事件 + 高频关键词

两种方式都不会输出运营建议/增长建议，只做客观信息归纳。

---

### 2. 新增「重要度排序」与「信号强度标签」— 解决"看不出哪些重要"

以前条目按时间排序，现在改为**按信号强度排序**：

```
重要度 = 来源数量 × 4 + 新鲜度加分
```

被多个独立来源同时报道的事件（说明这是真实、广泛的信号，不是个例噓声）
会排在最前面，并打上标签：

- 🔥 **多源印证**（≥3个来源同时报道）
- 📎 **2个来源**
- 无标签（单一来源，仅供参考）

这是典型的数据运营判断逻辑：**多源交叉验证的信号 > 单一来源的噪音**。

---

### 3. 改为「限高滚动」容器 — 解决"页面被拉得很长"

当某个分类的条目数超过 4 条时，卡片列表会被放进一个**限定高度、内部滚动**
的容器（约560px高，带细滚动条），而不是无限往下铺开。

效果：
- 页面整体高度可控，不会出现"刷不到底"的情况
- 想看更多详情，在该板块内部上下滚动即可，不影响整体页面结构
- 条目少的分类（≤4条）保持原样展开显示，不会强行套滚动框

---

### 4. 新增「环比变化」标签 — 解决"看不出趋势"

顶部分类总览的每个分类标签后面，新增了较昨日的增减箭头：

- ▲3（绿色）：较昨日新增 3 条
- ▼1（红色）：较昨日减少 1 条
- —（灰色）：与昨日持平

这是脚本自动读取 `docs/archive/` 里前一天的归档数据做比对，**首次运行时
没有昨天数据可比，不会显示**，从第二天起开始生效。

---

### 5. 总览标签变为可点击跳转

顶部 5 个分类标签现在是可点击的锚点链接，点击直接跳转到对应板块，
方便快速定位，不用整页滚动查找。

---

## 修复的一个潜在 bug

去重逻辑（`title_fingerprint`）原本只支持按空格分词，这对纯英文标题
（绝大多数自动抓取内容）没问题，但如果你在 `manual_inputs.csv` 里
手工填入中文标题，中文没有空格分词，会导致完全无法识别相似标题去重。
已修复为：检测到中文字符时自动切换为字符级 bigram 匹配。

## 不会改变的部分

- 仍然严格遵守"不输出增长运营建议"的要求——「今日要点」只做客观信息归纳
- 折叠区（失败来源、过滤信息、调试日志）保持原有结构，不占主要视觉空间
- 时效窗口、去重合并、来源展示等核心逻辑不变

2. 如果想要更智能的 AI 摘要，按 `README_无代码部署.md` 中「可选：开启 AI 智能摘要」章节配置 `ANTHROPIC_API_KEY`

## 追加更新：历史数据查询模块

新增「📅 历史数据查询」下拉框，位于面板顶部、总览标签上方。

**实现方式：**
- 每次运行写入 `docs/archive/{日期}.html` 和 `docs/archive/{日期}.json`（原有逻辑不变）
- 新增 `docs/archive/manifest.json`：自动扫描所有归档文件，生成「日期 + 当日条目数」索引
- 下拉框通过浏览器端 JS 读取这个 manifest，纯静态实现，GitHub Pages 原生支持，无需任何后端
- 在 `docs/index.html`（今日最新）和 `docs/archive/{日期}.html`（历史归档页）上都嵌入了同一个组件，
  两边的相对路径分别做了适配——历史归档页还会多出一个「↩ 返回今日最新」按钮

**使用方式：** 打开面板，顶部下拉框选日期 → 点「查看」→ 跳转到对应历史快照页面，
完整复现当天的「今日要点」「重要度排序」「来源列表」等所有内容，与首次发布时完全一致。

**限制：** 历史数据从功能上线当天开始累积，无法回溯到上线前的数据（之前没有写 manifest）。

## 追加更新：竞品监控矩阵 + 今日简报 + 风险标签

### 背景

用户反馈三个核心需求：① 真正想看的是"竞品的风吹草动"，而不是按来源类型分类；
② 看完信息后判断不出"这意味着什么"；③ 这个面板以后要给团队看，需要能快速扫读
和直接转发的摘要。

### 实现

**1. 品牌检测（`detect_brands` / `BRAND_REGISTRY`）**
每条合并要点的标题+摘要文本会被扫描，识别出提及了哪些已知竞品品牌
（NordVPN / ExpressVPN / Surfshark / ProtonVPN / Mullvad / Windscribe / PIA /
CyberGhost / IPVanish / X-VPN / AirVPN），结果挂在 `group["brands"]` 上。

**2. 情绪/严重程度判断（`classify_signal`）**
基于风险关键词库（断线/漏洞/诉讼/投诉/退款/欺诈等）和利好关键词库
（上线/获奖/审计通过/合作/升级等）做规则匹配，输出 `sentiment`（risk/positive/neutral）
和 `severity`（high/medium/low）。这一层始终是规则判断，不依赖AI，保证稳定。

**3. 竞品监控矩阵（`build_competitor_matrix`）**
跨越全部5个分类，按品牌汇总提及次数、风险信号数、利好信号数、代表事件，
按风险信号数降序排列，让你一眼看出"今天谁最该关注"。

**4. 今日简报（`generate_executive_brief`）**
顶部新增可复制的简报区块：先判断今日整体风险等级（🟢平稳/🟡需关注/🔴有预警），
再点出最值得关注的具体事件。AI模式下（配置`ANTHROPIC_API_KEY`）调用Claude
生成更精准的判断和解读；规则模式下用风险信号计数+Top品牌列表拼接而成。

**5. 卡片层风险标签**
每条信息卡片新增 🔴风险/🟢利好/🟡常规 标签和品牌标签，浏览单条信息时
也能立即看到判断结果，不需要等到看顶部简报才知道。

### 数据存储

`docs/data/latest.json` 和每日归档 JSON 中新增 `executive_brief`、
`competitor_matrix` 字段，以及每条要点的 `brands`、`signal` 字段，
方便未来如果要做更复杂的分析或对接其他工具。

## 修正：风险/利好判断未区分"我方"与"竞品"

### 问题

用户反馈："为什么NordVPN被投诉，对我们来说是风险？" —— 这是个真实的逻辑漏洞：
之前的版本只判断"这件事对该品牌自身是好消息还是坏消息"，完全没有考虑
"这个品牌是我们自己，还是竞品"。竞品出问题对我们通常是机会，不该标红色风险。

### 修正方案

新增 `config/own_brand.txt`（纯文本文件，GitHub网页可直接编辑，不需要写代码），
填入你自己的品牌名称（默认已预填 `X-VPN`，因为信源表里有标注这是自家产品）。

基于这个配置，新增 `compute_stance()` 函数，给每条信息打上更精细的"立场"标签：

| stance | 含义 | 显示 |
|---|---|---|
| `self_risk` | 我方品牌的负面信号 | 🔴 我方风险 |
| `self_positive` | 我方品牌的正面信号 | 🟢 我方利好 |
| `competitor_risk` | 竞品的负面信号 | 🎯 竞品负面·潜在机会 |
| `competitor_positive` | 竞品的正面信号 | ⚠️ 竞品正面·值得关注 |
| `brand_negative/positive` | 未配置我方品牌时的通用描述 | ⚠️/✅ 该品牌负面/正面信号 |
| `general` | 不涉及具体品牌 | 📰 行业信息 |

**竞品监控矩阵**也相应调整：我方品牌永远置顶（带⭐标识），新增"对我方意义"列
（需重点关注/潜在机会/竞品走强·留意/常规），并在表格下方加了一行说明文字，
明确"只有我方那一行的负面信号才代表风险"。

**今日简报**（AI模式和规则模式）也都更新为同样的立场区分逻辑，AI调用时会
明确告知"我方品牌是谁"，避免AI把竞品的坏消息也写成"我们的风险"。

**未配置 `own_brand.txt` 时**：系统会自动退化为"该品牌自身正负面"的中性描述，
不做相对我方的风险/机会判断（因为系统不知道"我方"是谁），并在简报和矩阵表格
里提示"配置 own_brand.txt 后可获得对我方意义的判断"。

## 视觉改版：参照 X-VPN 官网风格

### 背景

用户要求参照 xvpn.io 的外观重新设计面板视觉。

### 改动

整体从深色面板改为浅色、蓝紫渐变风格，呼应 X-VPN 官网的视觉语言：

- **配色**：背景改为浅灰蓝（#F5F7FC），卡片改为白色配柔和阴影（而非深色配边框），
  主色调改为 X-VPN 风格的蓝色（#2F5DFF）与紫色（#7C3AED）渐变

## 追加调整：宽屏多栏布局

### 背景

用户指出原先 960px 单列布局在宽屏显示器上浪费大量横向空间，5个分类从上到下
排成一条很长的单列，需要不停往下滚动才能看完。

### 改动

1. 页面容器最大宽度从 960px 放宽到 1440px
2. 5个分类区块改用 CSS Grid 自适应多栏布局（`grid-template-columns: repeat(auto-fit, minmax(380px, 1fr))`），
   根据实际可用宽度自动变成 1栏（手机）/ 2栏（平板/窄屏笔记本）/ 3栏（桌面宽屏），
   不需要手动设置断点，浏览器窗口缩放时会自动重新排列
3. 每个分类区块改为独立的卡片面板（白底、圆角、阴影），便于在多栏并排时
   彼此区分，视觉上更像仪表盘的多个面板而不是一条长列表
4. 「今日简报」「竞品监控矩阵」「历史查询」「分类总览」这几个横跨全局的模块
   仍保持整行宽度显示，不参与多栏布局（因为它们本身内容形态适合占满整行）

### 效果

桌面宽屏下，5个分类会自动排成 2-3 栏并排显示，大幅减少纵向滚动距离；
缩小窗口或在手机上查看时，会自动收窄为单栏，保持原有的移动端体验不变。

## 修正：社交媒体板块持续为空的根本原因

### 问题

用户反馈「社交媒体」板块始终抓不到信息。排查发现这不是脚本逻辑问题，
而是外部依赖（Nitter）本身在2026年已结构性失效：

1. 2023-2024年起，X/Twitter 封锁了 Nitter 用于匿名抓取的"访客账号"机制
2. 2026年现状更严峻：即便部分 Nitter 实例显示"在线"，**RSS 功能本身在多数
   存活实例上也已被关闭**——换实例地址也无法解决，因为问题出在"这条路本身被堵死了"

### 改动

1. **多实例轮询兜底**：新增 `NITTER_INSTANCES` 列表（nitter.net、xcancel.com、
   nitter.poast.org、nitter.privacyredirect.com 等6个实例），抓取时依次尝试，
   命中第一个可用的就停止。即便如此，根据当前外部环境，预期仍会大概率全部失败，
   这是诚实的预期管理，不是承诺修复
2. **推荐替代方案**：在 `REDDIT_提升指南.md` 新增「社交媒体板块抓不到信息」章节，
   推荐用 YouTube 官方 RSS 替代 Twitter 监控——100%稳定、免费、无需API Key，
   并给出无代码获取频道ID的具体步骤
3. `config/sources.csv` 中新增了 NordVPN/ExpressVPN/Surfshark 的 YouTube RSS
   模板行（默认 `enabled=false`，需要用户填入真实频道ID后启用）
4. `TROUBLESHOOTING_CN.md` 新增对应排错条目

### 诚实的预期管理

这次修复**没有让 Twitter 监控变得可靠**——这是外部环境的结构性问题，任何方案都
无法在不付费使用官方API（约$100+/月）的前提下完全解决。多实例轮询只是"死马当
活马医"的兜底，真正可靠的替代是 YouTube RSS（覆盖视频动态，覆盖不到纯文字推文）
或手工补充 `manual_inputs.csv`。

## 修正：今日要点/今日简报里出现未翻译的英文原文

### 问题

用户反馈「今日要点」希望看到中文总结，「今日简报」希望看到每个分类的总结。
排查后发现两个问题：

1. **规则模式（无AI Key）下，「今日要点」直接复述原始标题**——而绝大多数自动
   抓取的内容来自英文RSS源，所以没配置 `ANTHROPIC_API_KEY` 时，看到的是
   英文标题而不是真正的中文总结
2. **「今日简报」完全没有按分类拆开总结**——之前只讲跨分类的风险品牌，
   完全没有"5个分类各自在讲什么"这一层

### 改动

**1. 新增关键词→中文主题映射表**（`RISK_TOPIC_MAP` / `POSITIVE_TOPIC_MAP`）

不再直接复述英文标题，而是用结构化信号（品牌+主题+来源数）拼出真正的中文句子：

```
之前：NordVPN UK servers experiencing frequent disconnects（2个来源印证）
现在：NordVPN：连接故障（2个来源印证）
```

主题映射覆盖：数据泄露、法律诉讼、监管处罚/调查、合规问题、封禁限制、安全漏洞、
欺诈质疑、退款纠纷、用户投诉、连接故障（风险类）；审计/认证通过、获得奖项、
达成合作、新功能上线、功能升级、业务增长、获评推荐、运营成果（利好类）。

**2. 「今日简报」新增「分类速览」板块**，按 竞品动态/社交媒体/reddit讨论/
政策风险/第三方网站 固定顺序，每个分类输出一行中文总结（包含信息条数+
整体偏向+主要话题），没有数据的分类会明确写"暂无有效信息"而不是跳过：

```
—— 分类速览 ——
竞品动态：2条信息，偏正面（以新功能上线为主）
社交媒体：暂无有效信息
reddit讨论：4条信息，偏负面（以连接故障为主）
政策风险：1条信息，整体为中性常规动态
第三方网站：4条信息，偏负面（以退款纠纷为主）
```

AI模式下同样要求输出固定的5行分类总结，确保不会遗漏任何分类。

### 顺手修复的一个真实Bug

实现过程中发现关键词匹配存在子字符串误判：纯文本包含关系判断会把
"issues"（问题）误识别成包含"sue"（起诉）而错误标记为"法律诉讼"。
已改为英文关键词用单词边界正则匹配（中文关键词因无空格分词保持原有的
子串匹配），并补充了常见的英文单复数/时态变体（如 refund/refunds、
disconnect/disconnects/disconnected、complain/complains/complained），
避免严格边界匹配导致的"过严"漏判。

## 重大新增：增长信号周报（深度分析模块）

### 背景

用户反馈每日面板"还差点什么"——希望系统能帮忙找产品增长点，而不只是
监控"今天发生了什么"。通过逐步提问确认了具体边界：

1. 增长点关注三个方向：产品功能缺口、内容/营销角度、自身产品体验问题
2. 需要区分"偶发信号"和"反复出现的真实模式"，按证据强度分层展示
3. 不需要系统打分排序，只要把"反复出现的抱怨/需求"原始列出来，自己判断
4. 判定窗口定为30天，按月跑一次（实际定为每周一次，比月更及时）
5. 独立新页面呈现，与每日面板分开
6. 数据来源暂时只用面板现有来源；历史不足30天就用现有全部数据，不等

### 实现

**新文件 `scripts/growth_signals.py`**：独立脚本，不影响每日面板逻辑。
读取过去30天的归档JSON，用关键词主题表（`FEATURE_TOPIC_MAP`，覆盖流媒体解锁/
游戏延迟/分应用分流/客服响应/价格订阅/隐私政策等典型VPN产品维度）做聚类，
按品牌+情绪组合分到三个板块：

- 我方品牌+负面 → 自身体验问题
- 竞品+负面 → 产品功能缺口
- 不限品牌不限情绪 → 内容/营销角度（最宽的池子，与前两者有重叠，设计如此）

每个主题标注证据强度：🔥持续信号（≥3个不同日期出现）/ 💡观察中（少于3天）。
只做聚类排序，不打分、不给运营建议。

**新文件 `scripts/restore_archive.py`**：修复了一个之前埋下的真实问题——
GitHub Actions 每次运行都是全新检出，`docs/archive/` 历史文件从未持久化，
意味着day-over-day环比、历史日期下拉框、这次的30天趋势分析，理论上每次运行
都会发现"没有历史"。这个脚本在每次运行前，从当前已部署的 Pages 网站把历史
归档下载回本地，无需提交Git，自然解决持久化问题。已同步接入每日和每周两个
workflow。

**新文件 `.github/workflows/weekly-growth-signals.yml`**：每周一09:00 SGT
自动运行，也支持手动触发并自定义窗口天数。

**新文档 `GROWTH_SIGNALS_说明.md`**：完整使用说明，包括如何调整关键词主题表、
如何未来接入App Store/客服工单等数据源。

### 已知局限（设计上接受的权衡）

- 纯关键词聚类，不理解语义，可能有误判（如"slow"同时命中"连接速度"和
  "客服响应慢"两种语境）——可在 `FEATURE_TOPIC_MAP` 里手动调整关键词解决
- 不做语义级别的"机会评估"，只是原始聚类列表
- 历史数据从功能上线后才开始累积，无法回溯上线前的数据

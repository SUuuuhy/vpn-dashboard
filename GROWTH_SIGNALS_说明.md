# 增长信号周报 — 使用说明

## 这是什么

每周一次，把过去最多30天的归档数据按关键词主题聚类，分三个维度呈现：

| 板块 | 数据来源 | 回答的问题 |
|------|---------|-----------|
| 🔧 自身体验问题 | 我方品牌（`config/own_brand.txt`）相关的负面信号 | 我们自己反复被吐槽的点是什么？ |
| 🧩 产品功能缺口 | 竞品相关的负面信号 | 竞品反复被吐槽的点是什么？（可能是我们的机会） |
| 📢 内容/营销角度 | 所有信息，不限品牌不限情绪 | 哪些话题被反复讨论？（内容选题/关键词参考） |

**这不是一个"帮你做决定"的工具**——不打分、不排"机会大小"、不给运营建议。
就是把30天里反复出现的话题聚类列出来，按出现天数排序，你自己判断要不要跟进。

---

## 访问地址

```
https://你的用户名.github.io/vpn-dashboard/growth.html
```

也可以从每日面板顶部「📅 历史数据查询」那一行旁边，点「📈 增长信号周报」直接跳转。

---

## 运行频率

每周一 09:00（新加坡时间）自动运行一次。这是有意设计成"周"而不是"天"的——
判断"是否反复出现"本身就是个慢变量，没必要每天重算。

也可以手动触发：仓库 → Actions → Weekly Growth Signals Report → Run workflow。

---

## 证据强度怎么判断

- **🔥 持续信号**：同一个主题在过去30天内，出现在 **3个或以上**不同日期
  → 这是相对靠谱的"反复出现"信号
- **💡 观察中**：出现次数还不够多（1-2天）→ 可能是噪音，也可能是刚冒头的新趋势，
  仅供参考，不代表已验证

首次上线、或者历史数据还没攒够30天时，**直接用现有的全部归档数据跑**，
不会因为"不满30天"就空着不显示——只是这种情况下大部分主题会落在"观察中"，
随着每周积累，"持续信号"会逐渐增多。

---

## 关于"无法持续追踪历史"的修复

这个功能依赖读取过去30天的归档文件，但**GitHub Actions 每次运行都是全新的代码检出，
不会保留上一次运行生成的文件**。如果不做处理，每次都会发现"昨天的归档不存在"。

为了解决这个问题，新增了 `scripts/restore_archive.py`：在每次运行（每日面板更新、
每周增长信号）开始前，先从你**当前已经部署在线上的 GitHub Pages 网站**把过去的
归档文件下载回本地，再继续正常流程。这样不需要把归档文件提交回 Git 仓库
（保持仓库干净），也能让历史数据持续累积。

**这意味着**：30天窗口会随着时间自然填满——部署后第1周可能只有几天数据，
持续运行几周后会慢慢攒够30天。这是正常现象，不需要手动处理。

## 如何调整聚类的关键词主题

打开 `scripts/growth_signals.py`，找到 `FEATURE_TOPIC_MAP` 字典，
格式是 `"英文关键词": "中文主题名"`，按现有格式增删即可。

例如，如果你发现"slow"这个词同时被"连接速度慢"和"客服响应慢"两种场景误判
（这是关键词聚类的已知局限——纯粹按词匹配，不理解上下文），
可以把单独的 "slow" 拆分得更具体，比如改成只匹配 "connection slow"、
"speed slow" 这类更明确的短语，减少误判。

修改后 Commit + Push，下次周报运行时生效。

---

## 如何扩充"自身体验问题"的数据来源

当前只读取面板现有来源（Trustpilot X-VPN、社区讨论等）。如果以后要接入
App Store/Google Play 评论、内部客服工单等数据，思路是：

1. 把这些数据转换成跟 `config/manual_inputs.csv` 一样的格式（标题/摘要/时间/来源）
2. 通过 `manual_inputs.csv` 或新增一个类似的 CSV 文件喂给 `update_dashboard.py`
3. 只要这些条目能被打上 `own_brand` 的品牌标签，`growth_signals.py` 自动就会
   把它们纳入"自身体验问题"板块的聚类范围，不需要改 `growth_signals.py` 本身

---

## 三个板块为什么会有重叠

同一条信息可能同时出现在「内容/营销角度」和「产品功能缺口」（或「自身体验问题」）里——
这是设计如此，不是bug。「内容/营销角度」是最宽的池子（不限品牌不限情绪），
其他两个板块是更窄的子集（限定品牌+负面情绪）。三个板块回答的是不同问题，
不需要互斥。
